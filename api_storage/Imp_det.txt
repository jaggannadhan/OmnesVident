SUCCESSES
---------
1. Package structure
   api_storage/{__init__.py, database.py, repository.py, schemas.py,
   routes.py, tests/conftest.py, tests/test_repository.py,
   tests/test_routes.py}

2. Database schema (database.py)
   SQLModel (Pydantic + SQLAlchemy) with SQLite backend.
   Two tables:

   StoryRecord (story_record):
   - Flattened schema: all lead NewsEvent fields stored as direct columns
     (title, snippet, source_url, source_name, region_code, timestamp)
     for efficient SQL filtering without JSON parsing.
   - category, dedup_group_id indexed for O(log n) lookups.
   - mentioned_regions_json, secondary_sources_json: JSON string columns
     with Python property accessors for transparent List[str] serialization.
   - dedup_group_id has a UNIQUE constraint — enforces merge semantics at
     the DB layer, preventing duplicate rows from race conditions.

   SeenStory (seen_story):
   - Primary key: dedup_group_id (no surrogate key — direct lookup).
   - first_seen_at + last_seen_at enable both age-gating and audit.
   - Purged of entries older than 48 hours on app startup and during
     every ingest cycle (opportunistic cleanup).

3. Repository — insert path (repository.py)
   save_stories() iterates each EnrichedStory:
   - SELECT by dedup_group_id to determine insert vs merge.
   - INSERT: creates a new StoryRecord + SeenStory row.
   - Returns (inserted_count, merged_count) for observability.

4. Repository — upsert/merge path (repository.py)
   - If dedup_group_id already exists: merges incoming secondary_sources
     into the existing row, deduplicating URLs before write.
   - Updates SeenStory.last_seen_at on every merge.
   - Single session.commit() per batch (not per row) for performance.

5. Stateful deduplication — resolving Module 2 technical debt
   get_seen_group_ids(session, since_hours=24) returns the set of
   group IDs already processed in the last 24 hours.
   In POST /ingest: the Intelligence Pipeline's output is pre-filtered
   against this set before save_stories() is called — events whose
   dedup_group_id was already seen are silently dropped, preventing
   re-insertion of stories across ingestion cycles.

6. GET /news (routes.py)
   - Query params: region (ISO alpha-2), category, limit (1–200), offset.
   - Dual-region filter: matches region_code column (primary) OR
     mentioned_regions_json column (cross-regional).  A GB story about
     France appears under both /news?region=GB and /news?region=FR.
   - Response: PaginatedStoriesResponse {total, offset, limit, stories[]}.
   - Stories ordered by timestamp DESC.

7. GET /news/{region_code} (routes.py)
   - Path-param variant of the region filter.
   - Validates that region_code is exactly 2 characters (HTTP 422 otherwise).
   - Cross-regional discovery works identically to the query-param route.

8. POST /ingest (routes.py)
   - Accepts IngestRequest {events: List[NewsEventIn]}.
   - Returns 202 Accepted immediately.
   - Processing dispatched via FastAPI BackgroundTasks to keep the API
     non-blocking during Intelligence Pipeline execution.
   - Pipeline runs in asyncio thread-pool executor (run_in_executor) as
     it is synchronous CPU-bound work.

9. Async wrapper for IntelligencePipeline (routes.py)
   _process_and_store() calls:
     loop.run_in_executor(None, _pipeline.process, raw_events)
   This resolves the Module 2 technical debt item "pipeline.py is
   synchronous — needs async wrapper".

10. App lifespan (routes.py)
    On startup: create_db_and_tables() + cleanup_old_seen_stories(48h).
    Ensures the schema is always in sync and the SeenStory table stays lean.

11. Schemas (schemas.py)
    - NewsEventIn — wire schema for ingest input.
    - StoryOut — flattened API response (all array fields deserialized).
    - IngestResponse — {accepted, queued, message}.
    - PaginatedStoriesResponse — {total, offset, limit, stories[]}.

FAILURES / BLOCKERS
-------------------
1. Background task results not observable via the ingest endpoint
   POST /ingest returns 202 but provides no job ID or status URL.
   A caller cannot poll to confirm whether the batch was actually stored.
   A job-tracking table or WebSocket status endpoint is needed for
   production reliability.

2. get_latest_news total count is computed with a second full scan
   The current implementation executes the full query twice (once for
   total count, once with limit/offset).  For large datasets this is
   inefficient.  A SQLAlchemy func.count() subquery or window function
   should replace this in a later module.

3. Cross-regional filter uses JSON string LIKE matching
   mentioned_regions_json.contains('"JP"') is a substring match, not a
   proper JSON array query.  It works for MVP but could produce false
   positives if a region code appears as a substring of another value
   (unlikely with 2-char codes, but not guaranteed).  SQLite's json_each()
   or a normalised junction table would be more robust.

4. No authentication or rate limiting on POST /ingest
   The ingest endpoint is marked "Internal" but has no API key check,
   IP allowlist, or rate limit.  Any caller can trigger processing.


TECHNICAL DEBT
--------------
1. SQLite not suitable beyond MVP
   SQLite has no concurrent writers and limited JSON query support.
   Migration path to PostgreSQL: change DATABASE_URL env var and swap
   check_same_thread arg — SQLModel/SQLAlchemy handles the rest, but
   the JSON contains() filter (point 3 above) needs rewriting to use
   PostgreSQL's @> operator or a junction table.

2. StoryRecord.mentioned_regions_json uses property accessors
   The @property pattern works but SQLModel does not track changes to
   mutable JSON lists automatically.  All writes must go through the
   setter (currently enforced by convention, not the type system).

3. Cleanup is opportunistic, not scheduled
   cleanup_old_seen_stories() runs on startup and during ingest.  If
   no ingest occurs for > 48 hours the SeenStory table grows unbounded
   until the next restart.  A background scheduler (APScheduler or
   a cron endpoint) should run cleanup on a fixed interval.

4. No database migration toolchain
   Schema changes require dropping and recreating tables.  Alembic should
   be wired up before any production deployment or schema evolution.
