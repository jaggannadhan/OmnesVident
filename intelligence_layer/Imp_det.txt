  MODULE 2 — INTELLIGENCE LAYER: IMPLEMENTATION DETAILS
  Date: 2026-03-12
================================================================================

SUCCESSES
---------
1. Package structure
   intelligence_layer/{__init__.py, models.py, deduplicator.py,
   classifier.py, entities.py, pipeline.py, tests/}
   All specified files created.

2. EnrichedStory model (models.py)
   Pydantic v2 BaseModel extending the Module 1 NewsEvent:
   - lead_event: NewsEvent (the canonical, earliest-timestamped version)
   - secondary_sources: List[str] (URLs of duplicate articles)
   - category: str (one of 8 canonical labels)
   - mentioned_regions: List[str] (ISO alpha-2 codes)
   - dedup_group_id: str (8-char SHA-1 hex hash of title + region)
   - processed_at: datetime (UTC, ISO-8601 serialized)

3. De-duplication — Stage 1: Exact URL match (deduplicator.py)
   URL normalization strips scheme, www prefix, trailing slashes,
   query strings, and fragments before comparison.  O(1) set lookup.
   Confirmed correct with 3 dedicated URL normalization tests.

4. De-duplication — Stage 2: Fuzzy title match (deduplicator.py)
   - Library: thefuzz (Levenshtein distance via python-Levenshtein C extension)
   - Algorithm: fuzz.token_sort_ratio — order-insensitive, handles
     reworded headlines correctly (e.g. "Apple unveils X at event" ~
     "Apple unveils X at its annual event" → 96% similarity).
   - Threshold: 85 (configurable via FUZZY_THRESHOLD constant).
   - Guard: same region_code AND within 24-hour sliding window required.
   - Cluster lead: chronologically earliest event becomes the lead.
   - Tested: same-region merge, cross-region isolation, outside-window
     isolation, lead selection ordering.

5. Fuzzy threshold calibration notes:
   - At threshold=85, near-identical rephrasing merges cleanly.
   - "Market rally" vs "Market rally continues" scores 71 — correctly
     NOT merged (different stories, not just rewordings).
   - "Apple unveils iPhone 16 at annual keynote" vs same + "its" = 96 —
     correctly merged.
   - Threshold chosen to minimize false-positives over recall; can be
     tuned down to ~75 for higher recall if needed.

6. Topic classifier — keyword-based (classifier.py)
   8 categories: WORLD, POLITICS, TECHNOLOGY, BUSINESS, SCIENCE, HEALTH,
   ENTERTAINMENT, SPORTS.
   - ~130 keyword patterns compiled as whole-word case-insensitive regexes.
   - Title matches weighted ×2 (denser signal than snippet).
   - Highest-score wins; ties broken alphabetically for determinism.
   - Falls back to WORLD on zero matches.
   - 8/8 canonical category headlines classified correctly in parametrized
     tests.  No external model — sub-millisecond latency.

7. Geo-enrichment (entities.py)
   - 150+ country name entries (including adjective forms: "Russian",
     "French", "German", "British", etc.) mapped to ISO alpha-2.
   - Names sorted by length descending to prevent "Korea" matching before
     "North Korea" / "South Korea".
   - Primary region_code always guaranteed in output as first element.
   - Results de-duplicated and sorted for stable output.

8. IntelligencePipeline (pipeline.py)
   Chains: deduplicate → classify → extract_mentioned_regions.
   Accepts List[NewsEvent], returns List[EnrichedStory].
   Fully synchronous (all stages are CPU-bound).

9. Test suite — 41 tests, 0 failures, 0 warnings
   - test_deduplicator.py : 13 tests (URL normalization, exact match,
     fuzzy match, window gating, lead selection)
   - test_classifier.py   : 12 tests (8 category parametrized + 4 edge)
   - test_entities.py     :  8 tests (primary always present, adjective
     forms, multi-country, case-insensitive)
   - test_pipeline.py     : 10 tests (end-to-end EnrichedStory shape,
     dedup integration, category correctness)


FAILURES / BLOCKERS
-------------------
1. Classifier has no cross-category disambiguation training
   Keyword rules can produce ties on boundary stories (e.g., "Tech company
   reports record earnings" → both TECHNOLOGY and BUSINESS fire).
   Alphabetic tie-breaking is a stopgap, not a principled resolution.

2. Classifier has no recall for non-English text
   All keyword patterns are English-only.  Ingested stories in German,
   Japanese, Portuguese, etc. will almost always fall back to WORLD.

3. Geo-entity extraction misses demonyms and informal references
   "Washington announces..." (→ US) and "The Kremlin said..." (→ RU) are
   not covered.  A spaCy NER model or a larger gazetteer would be needed
   for production-level recall.

4. Deduplicator is O(n²) in the worst case
   Acceptable for batches ≤ ~500 events.  For higher-volume ingestion
   cycles, a locality-sensitive hashing (LSH) index should replace the
   nested loop.


TECHNICAL DEBT
--------------
1. Deduplication is stateless across ingestion cycles
   The current implementation deduplicates only within the current batch.
   A Redis or SQLite cache of (dedup_group_id, seen_at) is needed to
   suppress duplicates that arrive in different ingestion runs.

2. Classifier keyword list lives in source code
   Moving _RAW_RULES to a YAML/JSON config file would allow non-engineer
   tuning of categories without a code deploy.

3. Sliding window uses wall-clock timestamps from the article, not
   ingestion time.  Backdated articles (e.g., recirculated old stories)
   could bypass the window gate.  Ingestion timestamps should be tracked
   separately in a future data model version.

4. EnrichedStory.category is an unvalidated str
   A Pydantic Literal["WORLD", "POLITICS", ...] type would enforce the
   constraint at the schema boundary; currently only enforced by the
   classifier's output.

5. pipeline.py is synchronous — needs async wrapper
   When called from an async FastAPI endpoint (future Module 4), the
   pipeline should be wrapped in asyncio.get_running_loop().run_in_executor()
   to avoid blocking the event loop.

MODULE 16.0 — AI-AGENT GEO-REFINER
Updated: 2026-03-15
================================================================================

OBJECTIVE
---------
Replace the rule-based TagEnhancer geo-resolution path with an AI agent that
uses Gemini 1.5 Flash (Vertex AI) to accurately identify the geographic location
each story is ABOUT — not where it was sourced from.  Fixes the core "Mumbai
story tagged US-WA" problem caused by feed-geography ≠ story-geography.

IMPLEMENTATION STATUS: COMPLETE
---------------------------------

1. intelligence_layer/ai_geo_refiner.py  (new)

   Class: AIGeoRefiner
   Singleton: ai_geo_refiner (module-level)

   Key methods:
   - process_pending(limit=100) → int
       Main entry point.  Fetches pending raw_ingestion_buffer docs, processes
       them in batches of 15 through Gemini, promotes to master_news, marks
       buffer docs as "processed".  Returns count of promoted docs.

   - _call_gemini(stories) → List[dict]
       Sends a numbered batch prompt to Gemini 1.5 Flash via Vertex AI SDK.
       Runs sync SDK in asyncio thread executor (non-blocking).
       Parses JSON array from response; strips markdown fences if present.

   - _resolve(original, ai_result, client) → (region_code, lat, lng, confidence)
       Merges Gemini output with geo_lexicon cache:
         * If city name returned by Gemini is in geo_lexicon → use cached coords.
         * If city is new and Gemini has coords → populate geo_lexicon.
         * If confidence < 0.50 → keep original feed-level region_code.

   - _lexicon_get / _lexicon_set
       Read/write geo_lexicon Firestore collection keyed by city name (lowercase).

2. Gemini prompt design

   System prompt instructs the model to:
   - Identify the location the story IS ABOUT (not where it was published).
   - Return ISO 3166-2 subdivision codes (IN-MH, US-TX) when a city/state is named.
   - Use ISO alpha-2 country codes (IN, US) for national-level stories.
   - Set confidence < 0.5 for uncertain cases (keeps original region unchanged).
   - Return city centroid lat/lng, not country centroid.

   Output schema per story:
     {"region_code": "IN-MH", "lat": 19.07, "lng": 72.87,
      "city": "Mumbai", "confidence": 0.95}

   Example fix:
     Input:  title="Mumbai infrastructure plan unveiled for coastal highway"
             original_region=US-WA (from US tech feed)
     Output: region_code=IN-MH, lat=19.07, lng=72.87, confidence=0.95 ✓

3. Geo Lexicon (Firestore collection: geo_lexicon)

   Document ID: city name, lowercase (e.g. "mumbai", "chennai", "london")
   Fields:      region_code, lat, lng, confidence, _city, cached_at

   Purpose: avoid paying Gemini token cost for repeatedly-seen cities.
   After the first "Mumbai" story is resolved, all subsequent Mumbai stories
   skip the AI coordinate lookup and use the cached (19.07, 72.87).

   The lexicon grows automatically — no manual seeding required.

4. Promotion to master_news

   AIGeoRefiner calls mgr.promote_to_master(event, date_group, extra_fields)
   with extra_fields containing:
     - latitude / longitude   (AI-resolved city centroid)
     - geo_confidence         (Gemini confidence score, 0.0–1.0)
     - geo_source             "gemini-1.5-flash"
     - category               (rule-based classifier applied inline)

   Document ID remains SHA-256[:24](source_url) for idempotency.

5. /tasks/ingest update (api_storage/routes.py)

   Replaced the single FirestoreRefiner call with a two-stage approach:

   Stage 1: AIGeoRefiner.process_pending(limit=200)
     - AI-powered geo resolution + promotion.
     - If this promotes ≥ 1 doc, Stage 2 is skipped.

   Stage 2 (fallback): FirestoreRefiner.refine_pending(limit=200)
     - Rule-based TagEnhancer path.
     - Runs only when Stage 1 promotes 0 docs (Vertex AI quota exceeded,
       cold-start failure, or unavailable credentials).
     - Ensures raw buffer is never left stranded regardless of AI availability.

6. New dependency

   requirements.txt:
     google-cloud-aiplatform[generative_ai]>=1.70.0

   IAM: Cloud Run service account already has roles/aiplatform.user bound
   (same SA that has roles/datastore.user and secretmanager.secretAccessor).
   If not yet added:
     gcloud projects add-iam-policy-binding omnesvident \
       --member="serviceAccount:880643611795-compute@developer.gserviceaccount.com" \
       --role="roles/aiplatform.user"

VALIDATION
----------
Test case — "New bridge in Madurai":
  Input:  region_code=US (ingested from US international feed)
          title="New bridge inaugurated in Madurai"
  Expected Gemini output:
          {"region_code": "IN-TN", "lat": 9.93, "lng": 78.12,
           "city": "Madurai", "confidence": 0.92}
  Verify in Firestore master_news:

          geo_source  = "gemini-1.5-flash"

KNOWN GAPS / FUTURE WORK
-------------------------
1. Vertex AI free tier has limited quota; sustained 15-min cycles may hit
   rate limits.  Consider batching up to the quota ceiling or caching more
   aggressively in geo_lexicon.
2. Gemini occasionally returns fewer array items than the batch size (counted

3. geo_lexicon is append-only; stale coordinates (e.g. city renamed or

   job would improve long-term accuracy.
4. category assignment uses the rule-based keyword classifier, not Gemini.
